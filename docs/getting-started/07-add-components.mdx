A Bit component has all the information it needs for it to be **independently** shareable, consumable and maintainable. That includes its distributable code, development files, history, documentation and various Bit configurations.
The vast majority of that information is generated automatically by Bit. 

This section will start our component's journey to independency by creating its implementation files and have them tracked by Bit and handled, as a single component, by the workspace.

> The word “component” is ascribable to any independent feature, whether it is a simple UI primitive, a piece of logic, a data-connected component, or even a full page. 

## Create a UI Component

Let's create a UI 'Button' component (or clone the ['getting started' repository](https://github.com/teambit/getting-started-harmony) to skip this step)

```shell
$ mkdir -p components/react/ui/button  # Each Bit component have its own directory.
```

Now create the following files:

```shell
$ touch components/react/ui/button/index.ts           # Barrel file to bundle all exported modules.
$ touch components/react/ui/button/button.tsx         # Component's implementation (can be in multiple files).
$ touch components/react/ui/button/button.module.scss # Styling.
```

And implement the component as follows:

import Tabs from '@theme/Tabs';

<TabItem value="button">

```javascript
import React, { ButtonHTMLAttributes } from 'react';
import styles from './button.module.scss';
import cs from 'classnames';

export interface IButton extends ButtonHTMLAttributes<HTMLButtonElement> {
  /**
   * Determines whether button has a primary or secondary type of styling.
   */
  variant: 'primary' | 'secondary';
}

/** 
 *  A simple button component.
*/
export function Button({ children, variant, ...rest }: IButton) {
  return (
    <button className={cs(styles.base, styles[variant])} {...rest}>
      {children}
    </button>
  );
}

Button.defaultProps = {
  variant: 'primary',
};
```

## Add Dependencies

Our `Button` component uses `classnames`; to install it, run the following command:

```bash
$ bbit install classnames
```

Check the `workspace.jsonc` file and see that [`dependency-resolver`](/docs/dependencies/overview) section now contains the snippet below:

```json title="workspace.json"
"@teambit.bit/dependency-resolver": {
  "packageManager": "@teambit.bit/pnpm",
  "policy": {
    "dependencies": {
      "classnames": "^2.2.6"
    }
  }
}
```

As you recall from the ['Set up a workspace'](docs/getting-started/set-up-workspace) step, rules and policies set on the `workspace.jsonc` file are applied to all relevant components. The above ["policy"](/docs/dependencies/overview#dependency-policies) states that every component with 'classnames' as a dependency will use version 2.X.X.

You can use ['variants'](docs/variants/overview) to define specific policies for subsets of components.

## Track a Component

If you return to your workspace UI (in your browser at `localhost:3000`), you'll notice that our 'Button' component does not show up yet; this is because it is not tracked by Bit.

To track it we'll run the following command.

```shell
$ bbit add components/react/ui/button --namespace react/ui # Added component to Bit and set a namespace.
```

By adding a new component you trigger a process in Bit the does the following:

* Set component unique ID.
* Map a diretory to a component ID and logs to the `.bitmap` file.
* Map dependency graph using static code analysis.
